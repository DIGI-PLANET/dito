## RAG Architecture

```
User Message
    â†“
[1] Generate Embedding (Gemini text-embedding-004, 768 dims)
    â†“
[2] Store in diary_entries.embedding column
    â†“
[3] Similarity Search via match_diary_entries RPC
    - Cosine similarity > 0.7 threshold
    - Returns top 5 matching entries
    â†“
[4] Format as "Relevant Past Conversations" context
    â†“
[5] Inject into system prompt before AI generation
    â†“
[6] AI generates response with full context awareness
```

The embedding is generated **only for the user's current message**. Assistant responses are stored without embeddings. Only user messages with embeddings are searchable via RAG.

## Migration Logic (localStorage â†’ Supabase)

**File:** `src/lib/migrate.ts`

**Trigger:** Called when wallet connects (presumably from wallet-provider).

**Flow:**
1. Check `dito-migrated-{wallet}` flag in localStorage â†’ skip if set
2. Check if Supabase already has diary data for this wallet â†’ skip if yes
3. Migrate profile (display_name, language, ember_stage, interests, etc.)
4. Migrate all diary entries (iterate localStorage `dito-diary-*` keys)
5. Migrate all souls
6. Set `dito-migrated-{wallet}` flag
7. On failure: don't set flag â†’ retries next time

**Store hybrid pattern:** Every async store function (e.g., `getProfileAsync`) tries Supabase first, falls back to localStorage on error.

## Security Policies

### RLS
All 5 tables have RLS **enabled** but current policies are permissive (`USING (true)`). See [SECURITY-AUDIT.md](./SECURITY-AUDIT.md) for details.

### Service Role Usage
All API routes use `createServiceClient()` which creates a Supabase client with `SUPABASE_SERVICE_ROLE_KEY`, bypassing RLS. This is the intended pattern â€” API routes act as a trusted backend.

### Auth Model
**Wallet-address-based, no cryptographic verification.** The `wallet_address` parameter is trusted as-is. No signature verification is performed. This is a known MVP limitation.

## Groq Fallback Mechanism

**File:** `/api/chat/route.ts`

```
try {
  // Primary: Gemini 2.0 Flash
  const result = await genAI.generateContent(...)
} catch (geminiError) {
  // Fallback: Groq Llama 3.3 70B
  if (!process.env.GROQ_API_KEY) throw geminiError;
  const groqResult = await groq.chat.completions.create({
    model: 'llama-3.3-70b-versatile',
    ...
  })
}
```

- Gemini is the primary model (2.0 Flash)
- If Gemini fails (rate limit, outage, error), catches the error
- If `GROQ_API_KEY` is set, falls back to Groq's Llama 3.3 70B
- If no Groq key, re-throws â†’ returns 500 with fallback message
- Same system prompt and chat history are passed to both models
- Groq uses `max_tokens: 1024, temperature: 0.8`
- The soul generation route (`/api/soul`) does **not** have Groq fallback â€” Gemini-only
